Input

  • Brief — design a repo for many reports over time; start with a banking report; each report keeps its own folder, dataset, document & dictionary.
  • Starting point — an empty repo (just .claude/) on Windows, under a non-ASCII path.
  • Tooling — skills-for-fabric + Power BI agentic plugins (pbip / tmdl / pbir / semantic-models / reports).
  • Real data (mid-session) — a BankDIAD sample pasted in: Transactions, NPS, Targets, balances & dimension workbooks, monthly 2023–2025.

  • Report docs — dataset contract, data dictionary (md + csv), data profile, model design, build log.
  • Semantic model — TMDL Import model: 3 facts, 6 dimensions, 22 measures, 11 relationships, parameterised data folder.
  • Report — 3 pages / 31 visuals: Overview, Channel Performance, Branch Scorecard. Opens & validates in Desktop.

Six phases — a genuine sequence, each gated by a check or a decision.
1phase
Research

Read the skills-for-fabric structure and recommended flow (discover → Gold → Direct Lake → report), plus the exact PBIP / TMDL / PBIR rules from the installed plugins.

2phase

One folder per report under reports/NN-name/ with its own dataset / dictionary / docs / PBIP. Added conventions, a shared brand theme, templates and a registry.

3phase

Built a domain-assumption star schema and a minimal Overview page as a starting point — then paused at the right question: profile the real data before committing?

4phase
Profile & reconcile

Rebuild the model to the real schema

Profiled the pasted BankDIAD files, then rebuilt the model to match reality: Transactions / NPS / Branch Targets facts + real dimensions. Balance & loan tables reserved for future reports.

5phase

The longest phase. A chain of errors that only Desktop's strict parser raised — every lenient CLI validator said "fine". See the debug log below.

6phase
Confirm & build

Design-sign-off, then Pages 2 & 3

Confirmed business purpose, story and layout first, then built Channel Performance (12 visuals) and Branch Scorecard (11 visuals, incl. a per-branch matrix).

The debug log

Every blocker hit during Phase 5 — what Desktop said, why, and the fix. All four were invisible to tmdl-validate, pbir validate & validate_pbip.
desktop-load.recorder · 4 blockers · resolved RESOLVED
TMDL Format Error: InvalidLineType — Unexpected line type: Empty! · ./relationships, line 2
Root cause
A /// description followed by a blank line.

TMDL requires a /// description to sit immediately above the object it documents.

Fix
Removed the orphaned header.

Audited all 14 TMDL files for the same pattern; normalised to LF + UTF-8, no BOM.

TMDL Format Error: InvalidLineType — Unexpected line type: Other! · ./expressions, line 1 · '// Absolute path…'
Root cause
// comments aren't valid at the TMDL root.

Introduced while fixing the first error — the parser rejects bare comments outside an expression body.

Fix
Stripped all root-level comments.

Descriptions live in the data dictionary instead. Rule recorded in CLAUDE.md.

Required property 'reportVersionAtImport' was not included in /themeCollection/baseTheme & /customTheme of report.json
Root cause
Theme entries missing a required property.

Desktop rejected report.json, which cascaded into a misleading "no pages / ActivePageName not found".

Fix
Added reportVersionAtImport to both themes.

The page errors disappeared — they were a side-effect, not a real page problem.

ActivePageName not found · The report has no pages. Reports must have at least one page.
Root cause (the real one)
Page folders named Overview.Page.

PBIR folder names must be word-characters only. The dot makes them non-compliant, so Desktop silently ignores the folder — while every CLI tool "saw" 3 pages.

Fix
Renamed to Overview (no suffix).

Confirmed against Microsoft's PBIR docs. The breakthrough fix of the session.

F

Lessons & what to improve

What we'd tell the next person building a PBIP report with an agent.

◆ Learned

  • Profile real data before you model.The domain-assumption schema didn't survive contact with the BankDIAD export. The first model was a proposal, not the answer.
  • CLI validators are lenient; Desktop is the judge.tmdl-validate, pbir validate and validate_pbip all passed while Desktop still refused to open the file. Test in Desktop early.
  • "No pages" usually isn't a page problem.It was a non-compliant folder name (a dotted suffix) that Desktop silently ignores — and once, a cascade from a bad report.json theme.
  • Paste the exact error every time.Document + line number + line made each fix surgical instead of guesswork.
  • Environment bites.The non-ASCII Windows path triggered false positives from a plugin hook — wasted signal until identified.

▲ Improve next time

  • Keep a known-good PBIP template.Clone a report that already obeys every Desktop rule (folder names, LF/no-BOM, reportVersionAtImport) instead of hand-authoring from scratch.
  • Add a "Desktop-strict" pre-commit lint.Catch root comments, dotted page folders, missing theme props and BOMs before they reach Desktop.
  • Make profile-first a standard lifecycle step.Bake "paste data → profile → reconcile → build" into the report workflow doc.
  • Move the repo to an ASCII path.e.g. C:\src\… — removes the hook false positives entirely.
  • Close the open question.Define "revenue vs target" with the business; Revenue Target currently shows as context only.
