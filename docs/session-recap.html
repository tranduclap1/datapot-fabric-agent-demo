What we built

A new self-contained report folder in a Power BI / Microsoft Fabric monorepo — authored as diff-able PBIP (TMDL semantic model + PBIR report) and backed by a full set of analyst deliverables. The classic VanArsdel sales sample (100% synthetic): ~675K unit-level sales for 2015–2020 plus a 2020–2021 budget/forecast plan.

Input
  • VanArsdel_Actuals.xlsx (32 MB) — Date · Campaign · Customer · Product · Geo · Sales
  • VanArsdel_Budget_Forecast.xlsx — wide budget/forecast sheet
  • Repo conventions, shared templates & the Datapot theme
  • The brief: Part 1 + Part 2 deliverables, then build the report
Process
  • Profile-first ground truth before any modeling
  • Two multi-agent workflows (author + verify, then remediate)
  • Hand-authored TMDL; code-generated PBIR
  • Engine-level validation + post-open fixes
Output
  • Part 1: business glossary · data dictionary (+CSV) · data gaps & questions
  • Part 2: BRD · report spec · wireframe
  • Support: data profile · model design · dataset contract · build log
  • PBIP: semantic model + 5-page report, Datapot theme — opens & refreshes in Desktop

The process, step by step

P1Recon & gate
Read the repo + skills-for-fabric structure/workflow; confirmed it is built for multi-report development; then waited for the dataset before any data work.
P2Scaffold
Created reports/01-VanArsdel-analytics/ mirroring the repo's report structure (dataset · dictionary · docs · pbip) from the shared templates.
P3Profile the data ground truth
pandas/openpyxl: 675,368 sales rows, 0 orphan keys, and the findings that shaped everything — date sheet misses 16.6% of rows, Units is always 1, no revenue column (derive from price/cost), a uniform 27% margin, and an actuals∕budget overlap of only Jan–Jun 2020.
P4Author Part 1 & Part 2 workflow · 14 agents
Fanned out 7 documents from a single canonical backbone (one source of truth for names, DAX, numbers), then adversarially verified each against the profile. 3/7 clean, 4 flagged.
P5Remediate workflow · 11 agents
Fixed the verifier findings (measure count 21→20, project-name drift, a Device-vs-Channel mix-up, a broken ASCII schema diagram) and re-verified the four majors.
P6Build the semantic model
Hand-authored TMDL to the repo's conventions: 2 facts + 6 dims (calculated 2015–2021 calendar, conformed Product Group, snowflake Customer→Geo), 8 relationships, 20 measures.
P7Build the report
A deterministic Python generator emitted perfectly-formed PBIR — 5 pages, 43 visuals — from a single vetted PBIR schema. No hand-typed JSON, no agent drift.
P8Validate without the CLIs
The repo's validators weren't installed, so we used Power BI's own TOM library (model parsed clean — the real "will it open" test) + a Power Query replay in pandas (0 orphans, measures reconcile) + a DAX/binding check.
P9Post-open fixes after Desktop refresh
Real issues surfaced on first open: the Budget-vs-Actual page read a misleading 295.8% → fixed with a page filter to the valid overlap (97.7%); explained the flat 27% margin; improved slicers; re-synced the theme; reconciled every doc to the as-built report.

Efficiency move
43 report visuals were written by a ~250-line Python generator, not by the model.

Generating repetitive PBIR JSON in code instead of as LLM output saved a large amount of tokens and removed an entire class of hand-typing errors.

06

Lessons learned & what to improve

What worked / what bit us

Profile before you model

The date gap, Units≡1, the missing revenue column, the uniform margin and the budget overlap all came from profiling — each one changed the model. Domain assumptions would have shipped a wrong model.

Canonical backbone + adversarial verify

Giving 7 doc-writers one shared source of truth (names · DAX · numbers) and then a skeptical verifier per doc kept the set consistent and caught real drift before a human saw it.

Generate repetitive artifacts in code

A deterministic Python generator for 45 PBIR visuals beat hand-writing JSON — faster, cheaper, and drift-free.

Structure ≠ correctness

The 295.8% Budget bug passed every structural check. Only running the measures against the data exposed it. Verify visuals against the numbers, not just the schema.

Spec ↔ build drift erodes trust

The first docs promised maps and a scatter the build didn't include. Docs that describe what wasn't built are worse than terse ones — keep them in lock-step.

Validate with the real engine

With the repo CLIs missing, TOM (the parser Desktop itself uses) + a Power Query replay was far stronger evidence than JSON-shape checks.

Suggestions for next time

  1. Install the official PBIP CLIs (tmdl-validate, pbir, validate_pbip.py) in the environment, plus a headless refresh harness, so validation is native — not a workaround.
  2. Add a data-sanity / default-filter check that runs the headline measures against the data. It would have flagged the 295.8% attainment automatically, before a human opened the file.
  3. Add a spec↔build reconciliation step (ideally in CI) that diffs the report spec / wireframe against the visuals actually present in the PBIR.
  4. Build a reusable PBIR visual-template library so the next report's generator starts from vetted card / line / matrix / slicer blocks.
  5. Promote the "profile → canonical backbone → author → verify" flow into a repo skill so every new report runs the same vetted pipeline.
  6. Treat the parameterized data path as a feature — keep the DataFolder placeholder committed; the evaluator sets it and refreshes.
